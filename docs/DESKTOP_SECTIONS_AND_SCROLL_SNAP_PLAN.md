# План: экранные секции (desktop) + scroll snap (desktop) + paging dots

> Плановый документ: не является текущим регламентом разработки. Перед реализацией перепроверьте актуальную структуру в `src/components/sections` и `docs/ARCHITECTURE.md`.

Цель: на **desktop** (начиная с `md`, т.е. ширина \(\ge 768px\)) сделать секции главной страницы “экранными” по высоте окна (с учетом липкой шапки), а прокрутку между ключевыми секциями — с мягким “прилипанием” (scroll-snap). На **мобилке** ничего насильно не выравниваем и snap выключаем.

Важно: ниже **нет кода** — это пошаговый чеклист/ТЗ для Cursor AI.

- **Шапка**: `src/components/common/Header.jsx` использует `h-20` → это **80px**.

## 1) Базовое решение по высоте секций (desktop-only)

### 1.1. Принцип

- На desktop делаем **`min-height`**, а не `height`.
  - Причина: контент секции может быть выше экрана (особенно `Portfolio`, `FAQ`, `Contacts`). `min-height` не обрежет контент и не создаст “ловушки скролла”.
- Вычитаем высоту шапки: **80px**.
- На мобилке базовое поведение оставляем “как есть” (никаких `100vh`/`dvh` обязательных высот).

### 1.2. Единый “токен” высоты шапки (рекомендуется)

Чтобы не держать `80` / `5rem` в нескольких местах:
- Завести **CSS-переменную** `--header-h: 80px` в `src/styles/index.css` (глобально).
- Параллельно (опционально) завести JS-константу `HEADER_HEIGHT_PX = 80` в отдельном файле (например `src/constants/layout.js`) и постепенно уйти от “магических чисел” в `Header.jsx` и `App.jsx`.

Это поможет синхронизировать:
- выравнивание секций по высоте,
- якорную прокрутку,
- scroll-snap,
- paging dots.

### 1.3. Какие секции нужно привести к desktop screen height

Обновить секции главной страницы:
- `src/components/sections/Hero.jsx`
- `src/components/sections/Advantages.jsx`
- `src/components/sections/Services.jsx`
- `src/components/sections/Results.jsx`
- `src/components/forms/Calculator.jsx` (в проекте есть, в `App.jsx` используется) — проверить и применить, если на главной тоже должна быть “экранной”
- `src/components/sections/Portfolio.jsx`
- `src/components/sections/Process.jsx`
- `src/components/sections/FAQ.jsx`
- `src/components/sections/Contacts.jsx`

Идея правки:
- Добавить на **`md+`** `min-height` примерно: “высота экрана минус шапка”.
- Ничего не добавлять в базовом (mobile) слое.

Примечание про `Hero`:
- Сейчас `Hero` делает “почти экран” на мобильном и **отключает** `min-h` на `md` (`md:min-h-0`).
- По требованию наоборот: **на desktop сделать экранным**, а на мобилке — не фиксировать.

### 1.4. Вертикальная компоновка (осторожно)

Если хочется “как слайды” (контент по центру), можно добавить на desktop:
- контейнер секции `md:flex md:flex-col md:justify-center`

Но это может поменять дизайн (особенно где сетки/карточки). Поэтому:
- сначала сделать только `min-height`,
- затем точечно выравнивать (например `Hero` центрировать, а `FAQ`/`Portfolio` оставить обычными).

## 2) Scroll Snap: только desktop и только на главной странице

### 2.1. Почему так, а не “глобально на html”

Если включить `scroll-snap-type` на `html` всегда — он затронет:
- `/catalog`
- `/admin`, `/admin/login`
- любые будущие страницы

Нам нужно: **snap только на главной** и только desktop.

### 2.2. Стратегия включения/выключения snap

Вариант A (рекомендую): **тогглить атрибут/класс на `document.documentElement` только когда смонтирована главная**.

Шаги:
- В `HomePage()` (`src/App.jsx`) добавить эффект, который:
  - на desktop (`matchMedia("(min-width: 768px)")`) добавляет на `<html>` атрибут (например `data-home-snap="1"`)
  - на unmount — убирает атрибут
  - на resize — пересчитывает (если перешли границу `md`, включить/выключить)

Далее в `src/styles/index.css`:
- описать правила scroll-snap **только** для `html[data-home-snap="1"]`.

Плюсы:
- snap не трогает другие роуты,
- не нужно создавать новый scroll container,
- меньше шанс сломать sticky header и модалки.

Вариант B: делать отдельный scroll-container (wrapper с `overflow-y-auto`) только для главной.
- Это сложнее и имеет побочные эффекты (позиционирование, модалки, `position: sticky`, блокировка скролла).
- Использовать только если вариант A по какой-то причине “не цепляет” в целевых браузерах.

### 2.3. Режим снапа: “мягкий” proximity

Выбранный режим:
- desktop: `scroll-snap-type: y proximity`

Почему:
- ощущается “премиально”,
- не перетягивает управление у пользователя,
- меньше конфликтов с длинными секциями.

### 2.4. Snap только для “экранных” ключевых секций

По вашему требованию снапаются только:
- `Hero`
- `Services`
- `Results`

Остальные секции (`FAQ`, `Contacts` и т.д.) **без snap-align**, чтобы не мешать длинному контенту.

Практически это означает:
- на этих трех секциях на desktop добавить `snap-align: start` (Tailwind: `md:snap-start`)
- дополнительно (опционально) “фиксировать остановку” (Tailwind: `md:snap-always`), но включать только если поведение нравится и не раздражает

### 2.5. Компенсация липкой шапки при снапе и якорях

Сделать так, чтобы при снапе/скролле секция не пряталась под шапку:
- На snap-секциях добавить `scroll-margin-top` ≈ 80px (Tailwind: `md:scroll-mt-20`).
- Плюс глобально для home-snap режима можно поставить `scroll-padding-top: 80px` на `html[data-home-snap="1"]`.

Важно: у вас уже есть ручные оффсеты `headerHeight = 80` в:
- `src/App.jsx` (обработка `location.hash`)
- `src/components/common/Header.jsx` (клик по якорям)

Улучшение (см. раздел 4): постепенно уйти от ручного `scrollTo` к `scrollIntoView` + `scroll-margin-top`, чтобы не дублировать логику.

## 3) Paging dots / “Next section” вместо автодокрутки

Требование: вместо автодокрутки сделать **индикатор секций** (и кнопку “следующая”), чтобы у пользователя был контроль. Показываем только на desktop.

### 3.1. Поведение

- На desktop (md+) справа (или слева) фиксированная панель:
  - точки по количеству snap-секций (3 штуки: Hero/Services/Results)
  - активная точка подсвечивается
  - кликом по точке — скроллим к секции
  - кнопка “вниз/следующая секция” — скроллим к следующей
  - на последней секции “вниз” скрывается/дизейблится
- На мобилке:
  - компонент не рендерится

### 3.2. Какие ID/якоря использовать

Сейчас `Services` уже имеет `id="services"`.

Нужно:
- добавить стабильные `id` для `Hero` и `Results` (например `hero` и `results`) — чтобы:
  - paging мог ссылаться на DOM-элементы,
  - можно было делать `/#hero`, `/#results` при необходимости.

### 3.3. Определение активной секции

Рекомендуемый способ:
- `IntersectionObserver` по списку этих трех секций.
- Порог видимости: примерно 50–70% (подобрать визуально), чтобы активная точка менялась “ожидаемо”.

Fallback:
- если `IntersectionObserver` недоступен (редко) — простой расчет по `getBoundingClientRect().top` в обработчике scroll.

### 3.4. Доступность и UX-ограничения

- Учитывать `prefers-reduced-motion`:
  - если включено — прокручивать без smooth.
- `aria-label` на кнопках/точках.
- Не перекрывать важные UI элементы (например плавающие кнопки/виджеты) — оставить отступы и `pointer-events` только на самой панели.

### 3.5. Где разместить

Создать компонент, например:
- `src/components/common/SectionPager.jsx`

Подключить:
- только на главной (`HomePage()` в `src/App.jsx`)
- только на desktop (внутри компонента или через условие по ширине)

## 4) Улучшения, которые стоит сделать “заодно”

### 4.1. Один источник правды для высоты шапки

Сейчас значение 80px “зашито” минимум в двух местах. Рекомендуется:
- вынести в `src/constants/layout.js` (JS)
- и/или в `:root { --header-h: 80px; }` (CSS)

Цель: изменение высоты шапки не требовало “поиска по проекту”.

### 4.2. Упростить логику якорей

Сейчас при клике/загрузке якоря вы вручную делаете `scrollTo(top - headerHeight)`.

Более чисто:
- поставить `scroll-margin-top` на секции с `id`,
- использовать `scrollIntoView` (и управлять только `behavior`).

Это уменьшит шанс конфликтов со snap.

### 4.3. Экранная высота: `100vh` vs `100dvh`

Т.к. вы **исключаете мобилку**, на desktop обычно достаточно `100vh`.

Опционально (если хотите максимально “как в нативе” на современных браузерах):
- на desktop использовать `100dvh` как первичный вариант и `100vh` как резервный.

## 5) Риски/углы, которые нужно проверить

- **Длинный контент**: если секция длиннее экрана — она все равно должна нормально скроллиться (поэтому `min-height`, не `height`).
- **Portfolio modal**: он блокирует скролл, добавляя `overflow-hidden` на `body`.
  - Если snap будет включаться через `<html data-home-snap>`, это обычно ок.
  - Если вы решите делать отдельный scroll-container (вариант B) — придется блокировать скролл контейнера, а не `body`.
- **Sticky header**: убедиться, что при снапе верх секции не уходит под шапку (см. `scroll-margin-top` / `scroll-padding-top`).
- **Роуты кроме главной**: snap не должен проявляться на `/catalog` и `/admin/*`.

## 6) Чеклист реализации (порядок работ для Cursor AI)

1) **Добавить токен высоты шапки**
   - `src/styles/index.css`: `--header-h: 80px` (и при желании scroll padding для режима home-snap).

2) **Desktop screen height для всех секций главной**
   - Во всех секциях из списка 1.3 добавить desktop-only `min-height` “экран минус шапка”.
   - `Hero.jsx`: убрать mobile-`min-h` и перенести “экранность” на `md+`.

3) **ID для paging / навигации**
   - `Hero.jsx`: добавить `id="hero"` (или другой стабильный, но использовать его везде одинаково).
   - `Results.jsx`: добавить `id="results"`.
   - Убедиться, что `Services.jsx` уже `id="services"` и он остается.

4) **Включение snap только на главной и только desktop**
   - `src/App.jsx` → `HomePage()`:
     - добавить эффект, который на desktop добавляет атрибут на `<html>`, на unmount удаляет.
   - `src/styles/index.css`:
     - добавить правила scroll-snap только при наличии этого атрибута и только на desktop.

5) **Snap-align только на Hero/Services/Results**
   - `Hero.jsx`, `Services.jsx`, `Results.jsx`:
     - добавить desktop `snap-start` и `scroll-mt-*` под шапку.
   - `FAQ.jsx`, `Contacts.jsx` и остальные:
     - НЕ добавлять `snap-start`.

6) **Paging dots / Next**
   - Создать `src/components/common/SectionPager.jsx`.
   - Подключить в `HomePage()` (рядом с секциями).
   - Реализовать:
     - список секций: `hero`, `services`, `results`
     - `IntersectionObserver` для активной секции
     - клики → прокрутка к секции
     - кнопка “следующая”
     - desktop-only рендер
     - `prefers-reduced-motion` → без smooth

7) **Прогон линтера**
   - `npm run lint`

8) **Ручная проверка в браузере**
   - desktop: плавный snap между 3 секциями; dots работают; высоты секций адекватны.
   - mobile: snap отсутствует, dots отсутствуют, прокрутка обычная.
   - `/catalog`, `/admin/*`: snap отсутствует.

- Desktop (\(\ge 768px\)):
  - каждая секция главной имеет `min-height` примерно “высота окна − 80px”;

  - paging отсутствует.

  - snap нигде не проявляется.
